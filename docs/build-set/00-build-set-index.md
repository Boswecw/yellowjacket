# YellowJacket V1.1 Build Set Index

**Date:** 2026-04-12  
**Timezone:** America/New_York  
**Status:** Build-set index  
**Purpose:** Provide the canonical multi-canvas build set for YellowJacket so implementation can begin from a disciplined, reviewable, architecture-first document set.

---

## 1. Build Set Intent

This build set exists to turn YellowJacket from a strong concept into a governable implementation target.

It is designed to support:
- architectural lock before coding sprawl
- contract-first implementation
- verification-first proving posture
- explicit runtime boundaries
- clean repo/module separation
- local-first operational discipline

This build set is intentionally split into multiple canvases so each document can act as a stable control surface.

---

## 2. Canonical Canvas Set

### Canvas 01 — Governance and Doctrine
Defines the governing law above YellowJacket and the doctrinal identity of YellowJacket itself.

### Canvas 02 — Runtime Architecture
Defines runtime topology, bounded contexts, dependency seams, execution model, status model, trust domains, and checkpoint/resume architecture.

### Canvas 03 — Security and Trust Boundaries
Defines trust boundaries, permission envelopes, capability restrictions, action law touchpoints, prompt hygiene, and retention/redaction posture.

### Canvas 04 — Contracts and Schemas
Defines the canonical artifact taxonomy, lifecycle states, provenance posture, checkpoint/resume artifacts, and schema family expectations.

### Canvas 05 — Verification and Proving
Defines proving gates, golden corpus rules, replay and drift expectations, baseline-comparison requirements, and required test classes.

### Canvas 06 — Repo and Module Blueprint
Defines the logical package layout, dependency direction law, registration posture, artifact storage classes, and ownership boundaries.

### Canvas 07 — Build Roadmap and Start Gate
Defines the order of implementation, what must be built first, what must not expand too early, and the exact v1.1 start gate.

---

## 3. Build Philosophy

YellowJacket v1.1 is not a free-running swarm runtime.

It is a:
- governed local workcell coordination runtime
- sequential-first orchestration layer
- candidate-artifact producer
- verification-aware runtime
- review-bound subsystem
- bounded coordinator over declared seams

YellowJacket must not become:
- a universal agent brain
- a hidden execution engine
- a canonical truth owner
- an unrestricted model router
- an ad hoc packet compiler
- a background autonomy layer

---

## 4. Cross-Cutting Locks

The whole build set assumes the following locks:
- local-first must create real standalone value
- cloud is additive, not repair
- candidate authority is not execution authority
- provenance must be structured, not implied
- degraded-state truth must always surface upward
- every workcell must justify itself against a simpler baseline
- every meaningful boundary must be enforced outside the model
- checkpoint/resume must be deliberate, not accidental
- bounded parallelism is allowed only when declared and explainable

---

## 5. First Proving Target

The first proving class is:
- editorial review workcells

The first proving workcell is:
- Scene Continuity Review

That proving target exists because it is:
- review-heavy
- low-side-effect
- useful to real applications
- verification-friendly
- a strong test of candidate-vs-authority discipline

---

## 6. How To Use This Build Set

Use the canvases in this order:
1. Governance and Doctrine
2. Runtime Architecture
3. Security and Trust Boundaries
4. Contracts and Schemas
5. Verification and Proving
6. Repo and Module Blueprint
7. Build Roadmap and Start Gate

Do not begin broad implementation from architecture alone.  
Contracts, verification posture, and repo/module boundaries must be treated as equal start-stage control surfaces.

---

## 7. Final Index Statement

This build set is the v1.1 implementation control surface for YellowJacket.

Its purpose is to let you build a serious governed local workcell runtime without letting orchestration, reasoning, governance, execution, and persistence collapse into the same layer.
