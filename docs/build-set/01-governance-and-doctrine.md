# YellowJacket V1.1 Governance and Doctrine

**Date:** 2026-04-12  
**Timezone:** America/New_York  
**Status:** Governance and doctrine lock  
**Purpose:** Define the governing layer above YellowJacket and the doctrinal identity YellowJacket must obey before implementation expands.

---

## 1. Governance Mission

Forge Local Runtime Governance exists to keep the local runtime stack:
- bounded
- local-first
- explicit in authority
- truthful about degraded state
- reusable across applications
- resistant to silent boundary swallowing

This governance layer is above YellowJacket.  
It does not execute workcells.  
It defines the law YellowJacket must operate under.

---

## 2. Governing Principles

1. **Local-first must be real.** Local runtime must provide meaningful value without cloud dependence.  
2. **Cloud is additive, not repair.** Cloud may widen capability, but may not secretly complete the local baseline.  
3. **Reasoning authority is not execution authority.** A system that can analyze is not automatically allowed to act.  
4. **Candidate artifacts are default.** Inferential outputs remain candidates until promoted through governed paths.  
5. **Every subsystem must have bounded ownership.** No role swallowing.  
6. **Degraded-state truth is mandatory.** Reduced readiness must be stated explicitly.  
7. **Shared runtime capability must remain reusable.** Backbone runtime layers must not quietly become app-specific infrastructure.  
8. **Authority must be legible in architecture and UX.** The operator must be able to see what is candidate, governed, blocked, degraded, or review-bound.  
9. **Provenance and verification truth must travel with evidence.**  
10. **Hardware limits are architectural reality.** They must shape workcell design and runtime posture.

---

## 3. YellowJacket Core Identity

YellowJacket is the **governed local workcell coordination runtime** inside the Forge Local Runtime stack.

Its purpose is to coordinate bounded, reviewable, contract-driven multi-step local workcells by composing lower-level services without absorbing their authority.

YellowJacket is:
- coordination-first
- contract-first
- review-bound
- evidence-producing
- provenance-aware
- degraded-state explicit
- checkpoint-capable where allowed
- local-first by default

YellowJacket is not:
- the top governance layer
- the canonical truth owner
- the direct execution gate
- the persistence substrate
- the semantic file intelligence engine
- an unrestricted autonomous agent framework
- a replacement name for the whole architecture
- a model-serving subsystem
- a packet-compilation subsystem

---

## 4. Subsystem Allocation

### Governance layer owns
- doctrine
- authority law
- local/cloud posture law
- degraded-state vocabulary
- promotion law
- approval law interface
- trust-domain vocabulary

### YellowJacket owns
- bounded workcell orchestration
- role coordination within a workcell
- evidence assembly support
- verification-stage coordination
- review packet assembly support
- truthful run-level degraded-state disclosure
- checkpoint and resume coordination for admitted runs

### YellowJacket does not own
- canonical truth
- direct execution authority
- unrestricted filesystem or network power
- broad background autonomy
- hidden cross-run memory as default
- packet compilation responsibilities owned elsewhere
- model-serving policy internals

---

## 5. Core Doctrinal Laws

1. YellowJacket coordinates; it does not govern.  
2. YellowJacket may produce candidates; it may not silently execute.  
3. Every workcell must be explicit and bounded.  
4. Every workcell must run through declared contracts.  
5. Every workcell must declare degraded behavior.  
6. Every workcell must be inspectable.  
7. Every workcell must justify itself against a simpler baseline.  
8. Execution authority stays outside the free-running workcell.  
9. Provenance and verification status must be visible wherever evidence is consumed.  
10. Resource ceilings must be part of workcell truth, not hidden implementation detail.

---

## 6. Workcell Doctrine

YellowJacket hosts **workcells**, not generic conversational agents.

A valid YellowJacket workcell is:
- purpose-specific
- finite in scope
- schema-driven
- stop-condition bound
- evidence-generating
- provenance-carrying
- review-oriented
- candidate-output only unless explicitly handed off through governed paths

Every workcell must declare:
- entry contract
- dependency set
- role set
- degraded posture
- trust domain
- output classes
- review owner
- simpler baseline comparator
- checkpoint eligibility
- resource ceilings

A workcell must not:
- write canonical truth directly
- mutate state silently
- self-authorize tool expansion
- hide degraded conditions
- retain hidden memory across runs without declared policy

---

## 7. Candidate vs Authority Law

YellowJacket outputs are candidate artifacts by default.

This remains true even when outputs are:
- well structured
- useful
- internally consistent
- verified
- persuasive

Verification improves reviewability.  
It does not grant authority.

Authority remains outside YellowJacket for:
- canonical truth mutation
- effectful execution
- policy truth
- durable promotion

---

## 8. Degraded-State Law

Required state families:
- ready
- degraded
- unavailable
- blocked
- denied
- stale
- partial

YellowJacket must never suppress degraded conditions from:
- review packets
- verification decisions
- operator-visible summaries
- replay-supporting traces

No workcell may present cosmetic readiness when dependencies are degraded, unavailable, or stale.

---

## 9. Provenance Doctrine

Every workcell must distinguish between:
- source-derived evidence
- generated findings
- verifier-confirmed findings
- unverified model claims
- degraded or incomplete outputs

No evidence surface may blur these categories.

Provenance is not decoration.  
It is runtime truth required for review, replay, drift control, and trust.

---

## 10. Initial Proving Doctrine

YellowJacket proves itself first through review-heavy, low-side-effect local workflows.

The first proving class remains:
- editorial review workcells

The first proving workcell remains:
- Scene Continuity Review

YellowJacket expands only after proving:
- bounded usefulness
- verification discipline
- operator-legible review flow
- acceptable local performance
- lower governance burden than ad hoc orchestration

---

## 11. Final Doctrine Statement

YellowJacket v1.1 is the governed local workcell coordination runtime that composes bounded local services into reviewable candidate workflows with explicit provenance, truthful degraded-state disclosure, bounded resource posture, and checkpoint-capable local execution while keeping governance, execution authority, and canonical truth ownership outside the free-running workcell.
